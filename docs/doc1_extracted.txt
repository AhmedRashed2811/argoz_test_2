1. Executive Development Brief
The CRM must be built as a clean, flexible Django MVT application for one company first. The codebase must still be prepared for a future SaaS management application that controls CRM subscriptions, company isolation, integrations, and subscription-level settings. The current implementation should not overcomplicate the UI with SaaS screens, but the database and service boundaries must keep company scoping ready.
1.1 Non-Negotiable Development Rules
Rule
Technical Decision
Django MVT only
Frontend pages are rendered by Django templates. Use views, forms, templates, template tags, static JS/CSS, and standard Django URL reversing.
No frontend REST API
Do not build DRF endpoints for the template frontend. AJAX views may return JSON or HTML partials only for server-rendered UX behavior. External integrations may use webhook endpoints.
Custom authorization
Django auth may be used for login/session/password handling, but business permissions must be custom, DB-driven, page/action-aware, and editable from the UI.
No hardcoded authorization
Views, services, menus, buttons, and actions must check permission codes stored in the database, not Python role-name if/else logic.
Service layer first
Views must be thin: validate request, call service, return response. Business logic, distribution, campaign budget aggregation, SLA expiry, and permissions stay in services.
Audit everything
Every meaningful CRUD, permission change, policy change, assignment, stage change, approval, import, export, webhook event, background job result, and sensitive access must be audit logged.
OOP and DRY
Distribution techniques, policies, notification channels, approval logic, and integrations must be implemented through extensible classes/registries where possible.
Configurable policies
SLA times, distribution methods, reminder behavior, walk-in reception policy, source rules, campaign budget rules, and approval rules must be stored as configurable policies.
Relative URLs
All templates must use Django relative URL reversing, for example: {% url 'leads:detail' lead.id %}.

1.2 Business Scope Preserved in This Technical Design
Lead creation for Self-Generated, Campaign, Broker, Walk-in, Call Center, Exhibition, Referral, and Existing Client sources.
Direct vs Broker/Indirect lead classification, with configurable SLA assignment and breach handling.
Manual distribution, automatic Round Robin, automatic By Turn, language pre-check, team/salesman scope modes, retry attempts, and team escalation.
Lead stages: Fresh, Interested, Not Interested, Follow-up, Meeting, Not Reached, and Frozen, each with configurable SLA and reminder behavior.
Universal rule after SLA rotation: reset the visible lead stage to Fresh, reset Fresh SLA, preserve full history.
Marketing campaign master setup with multi-select campaign types: Events, TV Ads, Street Ads, Social Media, and Exhibition.
Campaign budgets, child budgets, Other Costs, finance approval, campaign lifecycle status, lead attribution, ROI, and KPI reporting.

2.3 One-Company Now, SaaS-Ready Later
Create a Company model and attach company_id to major business tables even though the first release has exactly one active company.
A default company resolver can return the single company during the one-company release. Later, the resolver can use subdomain, tenant slug, subscription context, or logged-in user membership.
All services should accept company or derive it from request context. This prevents a future migration from requiring a full rewrite.
Do not expose subscription billing UI now. Keep placeholders for subscription_status and plan_code if useful, but manage them later through the planned management application.

4. Custom Authentication and Authorization Engine
Critical Requirement: This is the most critical part of the system. Authentication can use Django login/session/password infrastructure. Authorization must be a custom CRM permission engine that non-technical admins can manage from the UI.

4.1 Authentication vs Authorization
Area
Decision
Authentication
Use Django User, sessions, password hashing, password reset, login/logout, account activation, and optional 2FA later.
Authorization
Use custom RoleGroup, PermissionDefinition, PageDefinition, RolePermissionDefault, UserPermissionOverride, and EffectivePermissionResolver.
Why not Django default permissions?
Django default add/change/delete/view permissions are too model-centric and not user-friendly enough for the required page/action/custom business permissions.
Django staff/superuser
Use only for internal technical admin access. CRM business access must not depend on is_superuser except for a restricted technical emergency account.

4.2 Permission Model Concept
A permission represents a business action or page/action ability, not only a database model action.
A page permission controls whether a user can open a page, menu, tab, or section.
An action permission controls what the user can do inside that page: create, edit, assign, approve, export, archive, upload asset, change stage, etc.
Every permission has a stable code such as leads.lead.view, leads.lead.assign_manual, marketing.campaign.budget.manage, finance.campaign.approve, or admin.users.create.
Roles are only default permission bundles. A user can receive a role, then lose some default permissions through explicit Deny overrides, and receive additional permissions from other roles through explicit Allow overrides.
Deny overrides win over Role defaults. Explicit Allow can add permissions from any module. This is what allows a Sales Manager to also have selected Marketing permissions without becoming a Marketing Manager.
4.3 Effective Permission Resolution Algorithm
Load all active roles assigned to the user.
Union all active default permissions allowed by these roles.
Load all direct user overrides: ALLOW and DENY.
Add all direct ALLOW permissions to the permission set.
Remove all direct DENY permissions from the permission set; Deny wins so admins can unselect default role permissions.
Apply optional contextual scope checks: own, team, branch, all-company, broker-owned, or campaign-owned.
Cache the result for the request and invalidate when roles, permissions, policy, or user overrides change.

4.5 View and Template Enforcement
# View-level authorization example
@login_required
@crm_permission_required('leads.lead.assign_manual')
def assign_lead_view(request, lead_id):
    form = ManualAssignmentForm(request.POST or None, company=request.company)
    if request.method == 'POST' and form.is_valid():
        LeadAssignmentService.assign_manually(
            lead_id=lead_id,
            actor=request.user,
            company=request.company,
            target=form.cleaned_data,
            request_meta=RequestMeta.from_request(request),
        )
        messages.success(request, 'Lead assigned successfully.')
        return redirect('leads:detail', lead_id=lead_id)
    return render(request, 'leads/assign_modal.html', {'form': form})

6.2 Audit Implementation Strategy
Use middleware to attach RequestMeta to a thread-local/context variable: user, company, IP, user-agent, request path, request method, request_id, and correlation_id.
Use model-level diff helpers for simple CRUD field changes. Use explicit service-level audit events for semantic actions such as assign, approve, rotate, reactivate, submit for finance, or webhook import.
Use transaction.on_commit to create notifications and async side effects only after the database transaction succeeds.
For sensitive fields such as phone/email, store raw before/after values only if the company accepts that audit policy; otherwise store masked values plus secure hashes.
Keep audit append-only. Never update audit rows except to attach archival metadata if needed.

8. Lead Management and Distribution Technical Design
8.1 Lead Creation Rules
Every lead requires name and phone number. Email is optional.
Phone number is validated on the frontend according to selected country code and revalidated on the backend.
The system asks whether the lead is Direct or Broker/Indirect. This affects SLA and reassignment behavior.
Every lead is Active by default unless created directly into an inactive terminal state by a controlled service action.
Duplicate detection must run before creating a new lead. If the existing lead is Active and within SLA, a “Call Me Again” / re-engagement case must be escalated to users authorized for manual distribution instead of automatic Round Robin.
8.2 Lead Source Rules
Lead Source
Business Rule
Technical Enforcement
Self-Generated - Sales Head
May assign to himself, to a salesman within his team, or run automatic distribution within his own team only.
Sales Head permissions + team scope.
Self-Generated - Salesman
Automatically assigned to himself. Policy decides whether it stays permanently with him or is redistributed after no-contact SLA.
Sales own-lead permissions.
Campaign
User selects campaign, then specific Event/Social channel/other campaign child. Distribution follows company policy: manual if authorized or automatic.
Campaign must exist and be active/selectable.
Broker
If broker logs in, lead is assigned to broker ownership first. Policy decides whether it is also assigned to company salesman. If another user enters it, the broker is selected and salesman assignment is optional per policy.
Broker ownership must be separate from salesman assignment.
Walk-in
Receptionist asks how visitor knew the company. Reception policy controls assignment: Open Floor, Team Turn, or Full Rotation.
Website must exist as an option in how-did-you-know lists.
Call Center
Call center user asks how caller knew the company. Distribution is manual or automatic based on policy.
Call center create permission.
Exhibition
User selects exhibition. Distribution is manual only and a salesman must be selected; lead cannot remain unassigned.
Exhibition/reference must be selected.
Referral
Referrer name captured as free text. Distribution manual or automatic according to policy.

8.3 Distribution Strategy Architecture
Technique
Business Logic
Implementation Contract
Manual Distribution
Authorized user assigns directly to a team or to a salesman. If assigned to team, Sales Head can decide who receives the lead.
ManualAssignmentService.assign_to_team() and assign_to_salesman(). Must create assignment history, audit event, SLA instance, and notification.
Round Robin
Load-balanced selection: first choose candidate with fewest active leads; tie-break by earliest last received lead date/time.
RoundRobinStrategy implements AssignmentStrategyInterface. Uses locking and candidate snapshot.
By Turn
Sequential fixed rotation across eligible pool. Skips inactive/unavailable recipients and advances pointer after successful assignment.
ByTurnStrategy implements same interface and uses RotationPointer model.
Retry Attempts + Team Escalation
Lead cycles through salesmen in current team for n attempts. After n attempts, escalates to next team and repeats.
RetryTeamEscalationService uses policy parameters, LeadRetryAttempt, TeamRotationPointer, and LeadAssignmentService.

8.4 Language Pre-Check and Scope Modes
Scope Mode
Rule
Language Pre-Check
Before automatic distribution, the system asks/selects lead language. Eligible pool is filtered to salesmen who support that language. Default language is Arabic unless policy says otherwise.
Mode 1 - Team, then Salesman
Automatic method selects a team, then selects a salesman inside that team.
Mode 2 - Team, then Sales Head Decides
Automatic method selects a team. Sales Head receives task/notification to manually choose salesman inside the team.
Mode 3 - All Salesmen

9.2 Universal Rotation Rule
Whenever a lead is rotated to a new salesman due to SLA expiry, the visible stage resets to Fresh.
The SLA clock restarts using the Fresh stage SLA configured by company policy.
All old stage history, notes, calls, meetings, attempts, and assignments are preserved.
The new salesman starts with a clear Fresh queue item and full SLA time, while management still sees the full audit trail.

10.1 Campaign Master

10.5 Campaign Lead Attribution and ROI
A lead generated from a campaign stores campaign_id.

Attendance can be captured through registration, reception entry, QR, landing page, manual entry, or webhook payload mapping.

12.2 SLA Job Concurrency
The SLA job must process batches and use database transactions.
Use select_for_update(skip_locked=True) on MariaDB so multiple workers do not rotate the same lead.

13. Meta Ads, Make/Zapier and Dynamic Webhook Architecture
13.1 Strategy
For the first release, use Make or Zapier as the middleware bridge for Meta lead ads. The CRM should not depend on a single global webhook. Instead, the system must create a tenant/company-specific webhook endpoint that can be copied into a Make/Zapier scenario. This keeps the setup self-service-ready for future SaaS clients while still working for the one-company implementation.
13.2 Dynamic Webhook Flow
Authorized admin or marketing manager opens Integration Setup.
User chooses provider: Make, Zapier, or future native integration.
CRM creates WebhookEndpoint with unique endpoint_uuid and secret token for the current company.
CRM displays the endpoint URL and setup instructions for Make/Zapier.
User configures Make/Zapier to receive Meta Lead Ads and send payload to CRM endpoint.
CRM receives payload, verifies token/secret, stores WebhookEvent, deduplicates event_id/hash, maps fields through WebhookMapping, then calls LeadCreationService.
LeadCreationService applies source rules, campaign attribution, language, policy-driven distribution, audit, and notifications.
Security / Reliability Feature
Implementation
Tenant-specific URL
/integrations/webhooks/<company_slug>/<endpoint_uuid>/ or UUID-only hidden endpoint linked to company.
Secret token
Require header or query token generated by CRM and stored hashed if possible.
Deduplication
Use provider event id when available; otherwise create dedupe_hash from normalized payload.
Mapping validation
Required fields: name and phone. Optional fields: email, campaign, platform, ad id, form id, language, source.
Idempotency
WebhookEvent status prevents duplicate processing. Lead duplicate service prevents duplicate CRM lead creation.
Retry
Failed event stored with error and retried by retry_failed_webhooks task.
Audit
Every received, processed, ignored, failed, and retried event creates audit/job logs.

14. Django MVT UI and Template Guidelines
Use base.html with navigation rendered from PageDefinition and filtered by EffectivePermissionResolver.
Every link must use Django URL reversing: {% url 'namespace:name' args %}. Do not hardcode internal paths in templates.
Use AJAX/fetch for modal forms, inline permission toggles, campaign child sections, budget recalculation previews, and notifications. AJAX endpoints are still Django views, not DRF APIs.
Use SweetAlert/confirm dialogs for destructive or high-risk actions: delete/archive, permission changes, finance rejection, SLA override, campaign approval.
Use server-side pagination/filtering for large lead/campaign/audit tables through standard Django views/selectors.
Hide buttons and table actions in templates when user lacks permission, but always enforce permission again in the view/service.
Use forms with dynamic sections for campaign type selection. The selected types determine which formsets appear.
Use user-friendly language in admin screens: display permission names/descriptions first and codes as secondary technical labels.

16.2 Service Transaction Pattern
@transaction.atomic
def assign_lead_to_salesman(*, lead_id, salesman_id, actor, company, reason, request_meta):
    lead = Lead.objects.select_for_update().get(id=lead_id, company=company)
    old_snapshot = LeadSnapshot.from_instance(lead)

    # Validate actor permission and business policy.
    PermissionService.require(actor, 'leads.distribution.manual_all', company=company)
    AssignmentPolicyValidator.validate_salesman_is_eligible(lead, salesman_id, company)

    # Apply mutation.
    lead.assigned_salesman_id = salesman_id
    lead.current_stage = LeadStageDefinition.objects.get(code='FRESH')
    lead.sla_deadline = SLAService.calculate_deadline(company, stage_code='FRESH')
    lead.save(update_fields=['assigned_salesman', 'current_stage', 'sla_deadline', 'updated_at'])

    LeadAssignmentHistory.objects.create(...)
    LeadStageHistory.objects.create(...)
    AuditService.log_change(actor=actor, before=old_snapshot, after=lead, request_meta=request_meta)

    transaction.on_commit(lambda: NotificationService.lead_assigned(lead.id, salesman_id))

17. Data Integrity, Concurrency and Validation Rules
Area
Required Protection
Campaign dates
Check end_date >= start_date in forms/services and DB CheckConstraint if possible.
Campaign type completion
Campaign cannot be submitted to finance until at least one type is selected and required child data for selected types is complete.
Budget values
DecimalField + MinValueValidator(0) + DB CheckConstraint for non-negative values. All writes go through CampaignBudgetService where possible.
Finance approval
Semi Approved and Not Approved require reason. Approval writes only through CampaignApprovalService.
Lead assignment concurrency
Use select_for_update on lead/SLA rows. Distribution candidate selection must lock relevant rows or record snapshot and revalidate before saving.
SLA jobs
Use select_for_update(skip_locked=True) on expired SLA instances. Idempotency key prevents duplicate rotation.
By Turn pointer
Lock RotationPointer row before reading/updating current_index.
Webhook events
Unique constraint on endpoint + external_event_id where available, or dedupe_hash.
Permission changes
Invalidate effective permission cache and write audit entry.
Broker ownership
Broker ownership and salesman assignment use separate fields and separate history tables.
Deletes
Prefer archive/soft-delete for business records to preserve audit and reporting.

20. Final Development Checklist
Django MVT pages render through templates; no frontend DRF dependency.
Every template URL uses {% url 'namespace:name' %}.
Every page and button is controlled by custom page/action permissions.
User creation wizard supports role defaults, unselecting defaults, and adding cross-role permissions.
Effective permission resolver supports role union, direct allow, direct deny, and scope filtering.
All business writes go through services, not direct view-created models.
Audit logs capture before/after values, actor, time, request metadata, and semantic action.
Lead duplicate and Call Me Again cases escalate to manual distribution users.
Broker ownership is separate from salesman assignment.
Round Robin and By Turn are strategy classes selected by code.
Language pre-check filters automatic distribution candidate pool.
SLA expiry supports Round Robin redistribution and Retry Attempts + Team Escalation.
SLA rotation resets stage to Fresh, resets Fresh SLA, and preserves history.
Follow-ups and meetings use services and create reminders/notifications/audit consistently.

Campaign type completion and budget validation are enforced before finance submission.
Budgets use DecimalField, non-negative validation, and snapshot history.

Celery Beat monitors SLA, reminders, webhook retries, email outbox, and campaign metrics.
